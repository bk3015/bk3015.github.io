Improving Johns Hopkins Course Review Querying

General Problem:

Locating old course reviews for classes at Johns Hopkins is a tedious and difficult experience. 

Primary Users:

Johns Hopkins University undergraduate and graduate students

Problem Details:

The inspiration for this report was because of the following issues with course review lookup:

  1. Locating the actual course review search is difficult and unintuitive. It must be located through a google search and then a JHU website linking to it.
  2. Old courses appear before new courses upon search, allowing potentially over a decade of courses that need to be sifted through
  3. Slight spelling or wording mistakes can make it tedious to find classes

Contextual Inquiry:

I conducted trials with two Johns Hopkins students to research user interaction on the site. One of the students is a senior and has used the course reviews feature extensively. The other is a sophomore and has never used course reviews.

  • Intent
    1. Discoverability – how easy is the course review search to find?
    2. Productivity – how long does it take to find a course’s information?
    3. Ease of use – how well does the interface augment the user’s action?

Key Takeaways:

  1. Discoverability
    1. If new to the service, it can be difficult to locate
    2. Even for experienced users, locating the site feels unintuitive
  2. Productivity
    1. For both experience and inexperienced users, the service is time consuming to use. One issue that contributes to this is the shear overload of potential course reviews. Additionally, when a professor teaches multiple sections of a class and there are then multiple course reviews from the same class, this creates confusion amoung the users.
  3. Ease of use
    1. When confronted with an incorrect input format, the new user failed to find the correct class. The more experianced user was about to give up but then found it right before.

“Paper-Based-Prototype”:

After reflecting on my initial hypothesis on how the issues with the current Johns Hopkins course reviews system work and combining that with the observations from the user studies, I determined the major focal point for my prototype should be simplicty. My first idea to adhear to this principle was actually the idea I ended up going with. A single search bar within SIS that prompts users with classes that are close to what they are searching. The user would then need to choose a course and their course review would then be displayed (no more need to download a PDF).

Review of Paper-Based-Prototype

After witnessing the test user use the paper base prototype here are the positive takeaways:

  • The new course reviews search functionality was designed to optimize simplicity which reduces the time needed to use the service
  • The new design only features a search bar where the user can search for classes by their name and/or by a professor’s name. The users can also choose the year they want to view the course reviews for.
  • The new course review's search functionality is within SIS to provide optimal locality for use cases.
  • The new design does not force the user to download a PDF.
  • The new design will suggest classes that are close to what the user is searching.

Negative takeaways:

  • Upon review with the user, they were discouraged that the new design did not allow them to search through multiple years at the same time
